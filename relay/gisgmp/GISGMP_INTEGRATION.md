# Интеграция ГИС ГМП → «Долги 1С» (полная схема для ИИ и людей)

> Документ-память по фиче: как реестр начислений **ГИС ГМП**
> (`gisgmp.cgu.mchs.ru`) подтягивается в ЖКХ-биллинг **utility-calc** и
> сверяется с выгрузками 1С (Excel ОСВ). Читать ПЕРЕД доработками — здесь все
> подводные камни, которые уже наступили.
>
> Прод: **asy-tk.ru** (auto-deploy при push в `main`). Контейнеры:
> `utility_calc_web_jkh`, `utility_nginx`.

---

## 1. Что это и зачем

Бухгалтерии нужно видеть **расхождения между долгами в 1С и в ГИС ГМП**
по каждому жильцу. ГИС ГМП — государственный реестр начислений (корп-сеть МЧС,
только из локалки). 1С — внутренняя бухгалтерия, выгружается в Excel (ОСВ) и
грузится в ЖКХ через парсеры во вкладке **«Долги 1С»** (раздел финансиста).

Фича добавляет к этому **автоматическую сторону ГИС ГМП**: серверный демон
(релей) на ВМ PODS2 логинится в реестр, собирает начисления, шлёт в ЖКХ; ЖКХ
кэширует их и **сверяет с последней Excel-выгрузкой 1С**, помечая проблемы.

**Ключ сопоставления — ФИО** (rapidfuzz, переиспользован из `gsheets_sync`).
Лицевые счета в ГИС и номера в ЖКХ не совпадают, поэтому матчим по имени.

---

## 2. Почему именно релей-демон (а не браузерное расширение / прямой запрос)

- `gisgmp.cgu.mchs.ru` доступен **только из корп-сети** (10.24.200.0/24 через
  Cisco-шлюз 10.23.0.1). Интернет-машины его не видят в принципе.
- **ВМ PODS2 — двудомная**: `ens19` = корп-сеть (10.23.1.183), `ens18` =
  интернет, + tailscale (100.119.4.25). Это **единственная** машина, которая
  видит И реестр, И `asy-tk.ru`. Поэтому сбор живёт на ней.
- Реестр за **OAuth2 (ORY Hydra) + passport.cgu.mchs.ru** (логин/пароль, НЕ ЭЦП).
- **Pull-модель**: релей раз в ~2 мин опрашивает ЖКХ (`relay-config`), а не ЖКХ
  пушит в релей. Так не нужно пробрасывать порт в корп-сеть; управление — из ЖКХ.

```
┌─────────────────┐  OAuth2/login   ┌──────────────────┐
│ gisgmp.cgu.mchs │◄────────────────│  ВМ PODS2 (релей)│  ens19 → корп-сеть
│  (реестр, корп) │  scrape charges │  /opt/gisgmp-relay│
└─────────────────┘────────────────►└────────┬─────────┘
                                              │ HTTPS + Bearer токен
                                              ▼ (ens18/интернет)
                                    ┌──────────────────────┐
                                    │  asy-tk.ru (ЖКХ)     │
                                    │  /api/financier/     │
                                    │   gisgmp/*           │
                                    │  кэш в SystemSetting │
                                    └──────────────────────┘
```

## 3. Поток данных (end-to-end)

1. **Релей** (`relay.py`, демон на ВМ) каждые `POLL_SECONDS` (120с) зовёт
   `GET /api/financier/gisgmp/relay-config`.
2. Конфиг отвечает: `enabled`, `months_back`, `should_run` (пора ли по
   расписанию/ручному запуску), `since` (курсор инкремента), `recheck`
   (очередь точечного дотягивания).
3. Если `recheck.surnames` непусто → релей **дотягивает** этих людей (см. §7).
   Иначе если `should_run` → **инкрементальный сбор** (см. §6).
4. Релей шлёт собранное в `POST /gisgmp/sync` (Bearer-токен).
5. ЖКХ (`gisgmp_import.sync_import_gisgmp_charges`) **мёржит по УИН** в кэш
   (`SystemSetting gisgmp_cache`), двигает курсор, **пересчитывает находки**
   (`gisgmp_findings`) из ПОЛНОГО кэша. **В таблицу показаний НЕ пишет.**
6. Финансист в ЖКХ открывает **«Сверка с 1С»** → `GET /gisgmp/reconcile`:
   агрегаты ГИС из находок × долги 1С из **последнего Excel-импорта** →
   на жильца флаг проблемы + severity. Можно **печатать** и **дотягивать**.

## 6. Доменная логика сбора (важно!)

**Структура строки начисления в реестре — 15 ячеек** (`<td>`), значимые в
`<div class="no-print">`:
```
[0]УИН [1]сумма [2]bill_date [3]actualize_date [4]КБК [5]ИНН [6]ОГРН
[7]КПП [8]лицевой_счёт [9]ФИО [10]назначение [11]ack_status
[12]change_status [13]source [14]uuid(в ссылке /charge/<uuid>)
```

**Счета:** `classify_account` по назначению/КБК → **209** (коммунальные услуги)
или **205** (наём). Долги считаются раздельно по 209 и 205.

**Что НЕ берём в долг:**
- `is_annulled` — аннулированные начисления (change_status). **Аннулированное
  не долг** (подтверждено пользователем на примерах).
- оплаченные (`is_unpaid` = False) — если квитанция сквитирована.

**Инкремент (сервер ГИС очень медленный/флапает 502/504):**
- сортировка `sort=c.actualizeDate&direction=desc` — новое/изменённое сверху;
- идём по страницам, **стоп** как только `actualize_date < since` (курсор);
- первый прогон (`since` пуст) — полный проход в пределах окна `months_back`;
- ретраи страниц при 5xx (`_fetch_page`, 3 попытки), ретраи push (4 попытки).

**⚠️ Курсор vs окно (грабли):** увеличение `months_back` ПОСЛЕ первого сбора
**не дотянет старую историю** — инкремент остановится на курсоре раньше, чем
дойдёт до старых дат. Чтобы углубить историю: либо сброс курсора + полный
проход, либо **точечное дотягивание** (§7) — оно ходит по `payerName` и не
зависит от курсора.

## 7. Сверка, авто-флаги и «Дотянуть расхождения»

### Источник сторон (после фикса «система видит старые файлы»)
- **ГИС-сторона** — агрегаты из `gisgmp_findings` (полный кэш, без аннулированных).
- **1С-сторона** — `applied_state` **ПОСЛЕДНЕГО завершённого Excel-импорта** по
  каждому счёту (`DebtImportLog.status=="completed"`, `file_name != "ГИС ГМП (авто)"`,
  сортировка по `started_at desc`). **НЕ** живые показания `MeterReading` —
  раньше туда писал ГИС-прогон и сверка показывала «старые цифры».
- `reconcile` отдаёт `source_1c={209:{file,at,log_id}, 205:{...}}` → UI/печать
  показывают, **какой именно файл** сверяется.

### Авто-флаги (анализатор сам сигналит проблемы)
На жильца (`eps`-допуск):
| Флаг | Условие | Смысл |
|---|---|---|
| `ok` | \|Δ\| ≤ eps | сошлось |
| `only_gis` | в 1С 0, в ГИС есть | «нет в 1С» — 1С не завёл |
| `only_1c` | в ГИС 0, в 1С есть | «нет в ГИС» — нет в реестре / другое ФИО / окно мало |
| `gis_more` | ГИС > 1С | лаг квитирования / 1С уже закрыл |
| `c1_more` | 1С > ГИС | окно ГИС мало (хроник глубже окна) |

**severity:** `high` ≥ 20000, `mid` ≥ 5000, иначе `low`. `problems[flag] =
{count, sum_abs, high}` — сводка для чипов. `matched_count` — сколько сошлось.

UI (`renderGisgmpReconcile`): чипы-категории = клик-фильтр (`this._reconFlag`),
поиск по фамилии, колонка «Флаг» с цветом, **⚠** на крупных. Лейблы/цвета в
`_RECON_LAB`. Кнопка **«Печать»** (`printGisgmpReconcile`) открывает чистый
печатный документ в новом окне (итоги + проблемы + таблица по жильцам) и зовёт
`window.print()` (печать или сохранение в PDF браузером).

### «Дотянуть расхождения» (умный пересбор — сошедшихся не трогаем)
`POST /recheck-build` берёт жильцов с флагами **`c1_more`** и **`only_1c`**
(там, где ГИС занижен), извлекает уникальные **фамилии** (первое слово ФИО) и
кладёт в `gisgmp_recheck`. `relay-config` отдаёт их и **гасит очередь** (claim).
Релей `do_recheck`: по каждой фамилии — `filtration[payerName]` + окно 36 мес,
все страницы (до 30) → доливает в кэш через `/sync` → пересчёт находок.
**Сошедшихся и `gis_more` не трогаем** — экономим тормозной сервер.
После — пересверка показывает, у кого ГИС подрос (стало лучше).

**Ключевой вывод по данным:** ГИС обычно показывает **БОЛЬШЕ** долга, чем 1С,
у хроников — ГИС копит всю неоплаченную историю, 1С усечён. Первый полный
сбор (окно 8 мес): 4516 начислений, 342 жильца, 284 сошлось, 58 не найдено.

## 7.1 Массовая актуализация (ГИС > 1С) и отслеживание результата

«**Актуализировать расхождения**» (`/gisgmp/actualize-build`) ставит в очередь
`gisgmp_actualize` неоплаченные счета жильцов с флагом **gis_more/only_gis** (ГИС
завышает) — релей (`do_actualize`) по каждому UUID дёргает в реестре
`GET /api/charge/{uuid}/actualize-request` (кнопка «Актуализировать из ГИС ГМП»).
Ответ реестра — `{"result":"…отправлен"}`, в истории начисления появляется
«Запрос на актуализацию статуса начисления → Отправлен в ГИС ГМП». Релей считает
`ok` только по этому `result`, а не по любому HTTP<400 (200 бывает страницей).

**⚠️ ГИС обрабатывает запрос АСИНХРОННО** («Отправлен в ГИС ГМП» → «Завершен» —
часы; реестр идёт в казначейство за статусом оплаты). Поэтому снимок «после»
**нельзя снимать сразу** — раньше `run_now` сразу после отправки давал вечное
«без изменений» (отсюда жалоба «ничего не актуализировалось»). Теперь:
- прогон проходит состояния **`running`** (релей шлёт actualize) → **`checking`**
  (авто-цикл) → **`done`**; поля `attempt`, `last_actualize_at`, `loop_result`;
- **авто-цикл доведения до «Сквитировано»** — `_drive_actualize_runs` (`relay-config`,
  ~каждые 2 мин): для `checking`-прогонов читает СВЕЖИЙ кэш, считает ещё несквитированные
  начисления прогона (по UIN, `is_unpaid && !is_annulled`), пишет «после»; всё
  сквитировано → `done(all_paid)`; каждые **30 мин** повтор актуализации несквитированных
  (макс **2 попытки**) → иначе `done(unpaid_left)` = реально не оплачено; **90 мин** —
  аварийный финал (`timeout`); каждый опрос ставит переопрос ФИО прогона (свежесть,
  реюз `do_recheck`); повтор — реюз `do_actualize` (та же очередь `gisgmp_actualize` +
  `run_id` прогона; `actualize-progress(finished)` → снова `checking`);
- **кнопка «Проверить результат»** (`/gisgmp/actualize-recheck`) — форсирует переопрос
  активных циклов (ускорить), статус не трогает;
- **по-персонно**: `/gisgmp/actualize-person` (несквитированные одного ФИО),
  `/gisgmp/person-charges` (список начислений человека — проваливание в «Сверке ФИО»);
- для **реального** долга `unpaid_left` после 2 попыток — норма (ГИС прав). ⚠️ Пока
  крутятся циклы, плановый сбор на паузе (у переопроса приоритет в цикле релея).

**Диагностика с сервера (без браузера):** `relay/gisgmp/probe_charge.py` логинится
кредами релея, печатает РЕАЛЬНЫЕ кнопки действий начисления (href + активна/откл) и
историю запросов; `--fire <path>` дёргает эндпоинт точечно. Так выяснили, что
релейный GET реально создаёт заявку. Забор на ВМ: `docker cp <web>:/app/relay/gisgmp/probe_charge.py /tmp/`.

Ключи `SystemSetting`: `gisgmp_actualize` (очередь+прогресс), `gisgmp_actualize_log`
(аудит прогонов «до/после», cap 50).

## 8. Развёртывание

### ЖКХ (utility-calc)
Push в `main` → GitHub Actions (self-hosted runner) → rsync → `docker compose`
→ миграции применяются авто. `.env` пишется из единственного секрета
`PROD_ENV_FILE`. Спец-действий не требуется; подождать ~5 мин сборку.

### Релей (ВМ PODS2) — ПЕРВАЯ установка
1. `relay.env` из `relay.env.example`: `GISGMP_SYNC_TOKEN`, `PASSPORT_USERNAME`,
   `PASSPORT_PASSWORD`, при нужде `MONTHS_BACK`/`POLL_SECONDS`.
2. `/etc/hosts` на ВМ (реестр по IP, т.к. корп-DNS не всегда):
   ```
   10.24.200.146 gisgmp.cgu.mchs.ru
   10.24.200.132 passport.cgu.mchs.ru
   10.24.200.130 hydra.cgu.mchs.ru
   ```
3. Маршрут в корп-сеть через Cisco (есть в `ExecStartPre` юнита):
   `ip route add 10.0.0.0/8 via 10.23.0.1 dev ens19`
4. `systemctl enable --now gisgmp-relay`; смотреть `journalctl -u gisgmp-relay -f`.

### Релей — ОБНОВЛЕНИЕ кода (после правок `relay.py`) — одна команда
```bash
TOKEN=$(sudo grep '^GISGMP_SYNC_TOKEN=' /opt/gisgmp-relay/relay.env | cut -d= -f2)
curl -fsS -H "Authorization: Bearer $TOKEN" https://asy-tk.ru/api/financier/gisgmp/relay.py \
  | sudo tee /opt/gisgmp-relay/relay.py >/dev/null
sudo systemctl restart gisgmp-relay
```
(self-update берёт `relay.py` из задеплоенного образа ЖКХ — сначала дождись
сборки ЖКХ, потом обновляй релей.)

## 9. Подводные камни (грабли, на которые уже наступили)

1. **`MeterReading` — ПАРТИЦИОННАЯ таблица, составной PK `(id, created_at)`.**
   `db.get(MeterReading, id)` падает; bulk-ORM-update молча НЕ работает на
   партициях. Только явный `update().where(MeterReading.id==...)`. (Сейчас
   сверка вообще не пишет в показания — но если будешь, помни это.)
2. **«Система видит старые эксели».** Причина: сверка читала живой
   `MeterReading`, куда писал старый ГИС-прогон. Фикс: 1С-сторона = последний
   Excel-`DebtImportLog.applied_state`; в ответе `source_1c` показывает файл+дату.
3. **Курсор vs окно** — см. §6: окно после первого сбора не углубляет историю.
   Для глубины — дотягивание по фамилии (§7).
4. **Сервер ГИС флапает** 502/504/таймауты → везде ретраи (`_fetch_page`, `push`).
5. **PowerShell + Python heredoc**: `& $py - @'...'@` передаёт heredoc как argv,
   python виснет на пустом stdin. Решение — писать скрипт во временный `.py`.
6. **Кириллица в git-сообщениях/консоли (Windows)**: коммит-месседж писать во
   временный UTF-8-без-BOM файл и `git commit -F`. Вывод анализа — в файл и Read.
7. **Сеть в песочнице** (`git push`/`curl`) — «Could not resolve host». Запускать
   с `dangerouslyDisableSandbox:true`.
8. **`SystemSetting.value`** — TEXT, всё JSON-сериализуем сами (`json.dumps`,
   `ensure_ascii=False`). Cap-ы (cache 20000, findings 8000) — чтобы не распухло.

## 10. Диагностика конкретного жильца (для будущих ИИ)

Чтобы проверить, что ГИС начисляет человеку (на ВМ, прямой запрос в реестр),
использовался скрипт `check_payer` (логин → `filtration[payerName]=Фамилия` →
сумма по 209/205, с учётом аннулирования). Примеры проверок (factual):
- Власенко: 209=32 667.04, 205=7 178.18
- Абдулмеджидов: 209=10 560.28, 205=1 622.31
- Тихонов: 209=3 048.15, 205=2 202.68

Если такой скрипт понадобится снова — он логинится тем же `login()` из
`relay.py`, фильтрует по `payerName`, парсит `parse_page`, суммирует `is_unpaid
and not is_annulled` по `classify_account`.
