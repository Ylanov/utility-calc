# Авто-подгрузка долгов/переплат из 1С (БГУ) через релей-браузер

> Фича в **utility-calc**. Тот же релей-демон, что ходит в ГИС ГМП
> (`/opt/gisgmp-relay/relay.py`), headless-браузером (Playwright) заходит в
> веб-клиент **1С:Бухгалтерия государственного учреждения 2.0** на
> `https://sv00web19.mchs.ru`, формирует **«Оборотно-сальдовую ведомость по
> счёту»** за период (с начала года → сегодня), сохраняет в Excel и шлёт в ЖКХ.
> Файл проходит через **тот же** парсер, что и ручная загрузка Excel-ОСВ.
>
> Прод ЖКХ: **asy-tk.ru** (auto-deploy при push в `main`).

---

## 1. Почему именно так

- `sv00web19.mchs.ru` — это **полноценный тонкий веб-клиент 1С 8.3.27** (маркеры
  `e1cib/*`, `seanceId`, `mod_main_mainform.js`), а НЕ простой опубликованный
  отчёт. Данные ОСВ рисуются динамически по протоколу `e1cib` (stateful RPC,
  недокументирован, привязан к версии) → **«реплей POST формы Сформировать» не
  работает**.
- OData/HTTP-сервис 1С — недоступен (админы 1С не включают). Поэтому путь —
  **браузер-автоматизация**: гоняем реальный веб-клиент как человек.
- Релей — единственная машина в корп-сети МЧС, которая видит и `sv00web19`, и
  `asy-tk.ru` (как для ГИС ГМП). Логика живёт на нём.
- **Максимальный реюз:** релей лишь воспроизводит ручной экспорт ОСВ в Excel;
  разбор/матчинг/гейт «Выгрузить» — существующий конвейер `debt_import.py`.
  Никакого парсинга нарисованной таблицы (хрупкого) — только готовый xlsx.

```
┌───────────────────────┐  Playwright   ┌────────────────────┐
│ sv00web19.mchs.ru     │◄──────────────│  релей (ВМ, корп)  │ headless Chromium
│ 1С:БГУ веб-клиент     │  логин→ОСВ→   │ /opt/gisgmp-relay  │
│ (ОСВ по счёту)        │  →xlsx        │  relay.py          │
└───────────────────────┘──────────────►└─────────┬──────────┘
                                                   │ HTTPS+Bearer (GISGMP_SYNC_TOKEN)
                                                   ▼
                                       ┌──────────────────────────────┐
                                       │ asy-tk.ru  /api/financier/    │
                                       │   onec/relay-config (creds)   │
                                       │   onec/sync (Excel→черновик)  │
                                       │  → sync_import_debts_process  │
                                       │  → ручная кнопка «Выгрузить»  │
                                       └──────────────────────────────┘
```

## 2. Поток данных (end-to-end)

1. Финансист в ЖКХ («Долги 1С» → карточка **«Авто-подгрузка из 1С»**) вводит
   **логин/пароль 1С** (шифруются Fernet, ключ `ENCRYPTION_KEY`), задаёт **счета**
   (наём → наш `205`, коммуналка → наш `209`) и час сбора, включает галку.
2. Релей раз в ~2 мин зовёт `GET /api/financier/onec/relay-config` (тот же
   `GISGMP_SYNC_TOKEN`). Ответ: `enabled`, `should_run`, `accounts` (код+тип),
   `period` (`01.01.<год>` → сегодня), `credentials` (отдаются **один раз**,
   claim `creds_pending`).
3. По `should_run` релей `run_onec`: headless-браузер → логин → **Стандартные
   отчёты → Оборотно-сальдовая ведомость по счёту** → для каждого счёта: задать
   счёт+даты → **Сформировать** → **сохранить в xlsx** (ловим браузерный download).
4. Релей `POST /api/financier/onec/sync` (multipart `file_205`/`file_209`).
5. ЖКХ сохраняет файлы и ставит **те же** Celery-задачи `import_debts_task`
   (`stage_only=True`) → **черновик** `DebtImportLog` (имя «1С ОСВ … (авто).xlsx»).
6. Финансист видит черновики → жмёт **«Выгрузить»** (`/debts/publish`) → долги/
   переплаты применяются жильцам. **Дебетовое сальдо = долг, кредитовое = переплата**
   (нетирование Дт−Кр уже в `debt_import.pick_saldo_pair`).

## 3. Эндпоинты (`/api/financier/onec/...`)

| Метод | Путь | Авторизация | Назначение |
|---|---|---|---|
| GET | `/onec/status` | финансист | статус для UI (без пароля) |
| PUT | `/onec/config` | финансист | enabled, счета, час, base_url, infobase |
| POST | `/onec/credentials` | финансист | логин/пароль 1С → шифр → `creds_pending` |
| POST | `/onec/run-now?probe=` | финансист | взвести запуск (probe=true — разведка) |
| GET | `/onec/relay-config` | Bearer | конфиг релею + creds (claim) + should_run |
| POST | `/onec/relay-report` | Bearer | релей отчитывается (статус/счётчики) |
| POST | `/onec/sync` | Bearer | Excel-ОСВ → черновик (reuse import_debts_task) |

Хранилище без миграций — `SystemSetting key="onec_relay"` (конфиг+статус+шифр).

## 4. Развёртывание релея (Playwright на ВМ)

Релей — тот же `/opt/gisgmp-relay`. Добавить браузер:
```bash
# в venv релея:
pip install playwright
python -m playwright install chromium
python -m playwright install-deps chromium   # системные libs (если есть sudo)
```
Код 1С едет внутри `relay.py` — обновляется штатным self-update («Обновить релей»
в UI, см. GISGMP_INTEGRATION.md §8) ПОСЛЕ деплоя ЖКХ. `playwright` импортируется
**лениво** — если не установлен, ГИС-ГМП-ветка работает, а 1С-ветка пишет ошибку
в статус.

## 5. Разведка-first (важно про селекторы)

Веб-клиент 1С даёт **динамические id** → точные селекторы пишем по реальному DOM.
Первый прогон — **«Разведка (probe)»** в UI: релей логинится и кладёт скрины+HTML
в `/opt/gisgmp-relay/onec_probe/` (`01_open`, `02_after_login`, `03_report_form`,
`desktop.html`, `report_form.html`). Их отправляют разработчику → он дорабатывает
`_onec_open_report` / `_onec_collect_account` / `_onec_save_xlsx` под реальную
разметку, бампает `RELAY_VERSION`, деплой → «Обновить релей».

Текущие селекторы (best-effort, до первой разведки):
- **логин**: якорь `input[type=password]`, логин — первое видимое текстовое поле,
  кнопка по тексту «Войти/ОК» или Enter.
- **навигация**: текстом «Учет и отчетность» → «Стандартные отчеты» → имя отчёта.
- **форма ОСВ**: поле по подписи «Счет», даты по «с»/«по», кнопка «Сформировать».
- **экспорт**: меню «Ещё»→«Сохранить как» → формат «Лист Excel 2007 (xlsx)» →
  ловим download. ⚠️ Если 1С не отдаёт браузерный download — переключимся на
  чтение нарисованной таблицы из DOM (запасной путь).

## 6. Подводные камни

1. **Файл должен быть .xlsx** (PK-zip) — `_save_uploaded_debt_file` проверяет
   magic-bytes. 1С «Лист Excel 2007». `.mxl` (родной 1С) не пройдёт.
2. **Счета настраиваются в UI**, не хардкод. На скрине у пользователя — `205.31`
   (наём). Коммуналку (`209.xx`) надо заполнить. `account_type` (`205`/`209`)
   определяет, как `sync_import_debts_process` классифицирует долг.
3. **Креды одноразовые** (`creds_pending` claim). Сменил в UI → снова уйдут.
   Релей хранит их в `onec.env` (переживает рестарт демона).
4. **Период** считается на сервере (МСК): `01.01.<год>` → сегодня. Меняется в
   `onec_relay_config`.
5. **Гейт ручной** — авто-сбор только готовит черновик; на жильцов ничего не
   попадёт без кнопки «Выгрузить». Контроль за финансистом.
